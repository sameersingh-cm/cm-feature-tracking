System overview

A scheduled automation system that tracks product scope decisions across three sources — Confluence PRD, Slack channels, and UAT Google Sheets — and writes structured, up-to-date output to a master Google Sheet. The system is resilient, observable, and fully config-driven.

Principle
Passive capture
No behavior change required from the team. Everything is read automatically.
Principle
Current state + full history
Section 1 & 2 are upserted. Changelog is strictly append-only.
Principle
Resilient pipelines
One source failing never blocks the others. All errors are logged.
Principle
Config-driven
All feature settings live in a Google Sheet. Nothing hardcoded.
Principle
Token-efficient
Scheduled batch runs only. No real-time event processing.
2
Functional requirements
FR-1: Config sheet management
  • System reads Config Google Sheet at the start of every run
  • Features with Active = N are completely skipped — zero API calls
  • System writes run completion status back to Config Sheet Run Log tab
FR-2: PRD ingestion (Confluence)
  • Connects to Confluence via Atlassian API, reads PRD page per active feature
  • Extracts structured feature list using Claude API, snapshots with timestamp
  • On subsequent runs, diffs against previous snapshot and logs deltas
  • If Confluence fetch fails → log error to Run Log, skip PRD, continue other pipelines
FR-3: UAT sheet ingestion (Google Sheets)
  • Identifies latest UAT Status column dynamically (e.g. UAT 1, 2, 3 → uses UAT 3)
  • Applies deterministic rules — no AI needed here, rules are reliable
  • Outputs disposition: Pushed / Parked / Active Blocker / No Action
  • Writes to Output Sheet (Section 1 upsert + changelog append)
FR-4: Slack ingestion
  • Reads full threads (not individual messages) from configured channels
  • Pre-filters by keyword before sending to AI — reduces cost and noise
  • All detections logged regardless of confidence score; confidence column visible for audit
  • Per-channel failure isolation — one channel failing doesn't stop others
FR-5: AI classification (Claude API)

FR-6: Output sheet writing

FR-7: Manual trigger

A dedicated HTTP endpoint that triggers a full pipeline run immediately. Returns a JSON summary. Callable from browser, curl, or any HTTP client.

FR-8: Feature on/off toggle

Active column in Config Sheet Tab 1. If Active = N, zero API calls are made for that feature. Toggle takes effect on the next run.

3
Non-functional requirements
Resilience
Each pipeline runs independently inside its own try/catch. Failure of one never blocks others.
Idempotency
Running the pipeline twice for the same window creates zero duplicate changelog entries. Source message IDs used as dedup keys.
Cost efficiency
Batch processing only. Slack messages pre-filtered by keyword before AI classification.
Auditability
Every run logged with full detail. Every AI decision logged with confidence score.
Maintainability
All configuration in Google Sheet. No hardcoded feature names, channel IDs, or URLs in code.

5.2 Orchestrator logic
async function runPipeline(triggerType = 'scheduled') {
  const runId = generateRunId()
  const runStart = new Date()
  const manifest = await configReader.load()
  const activeFeatures = manifest.filter(f => f.active)

  // All 3 pipelines run in parallel — Promise.allSettled ensures
  // one failure never stops the others
  const results = await Promise.allSettled([
    prdPipeline.run(activeFeatures),
    uatPipeline.run(activeFeatures),
    slackPipeline.run(activeFeatures)
  ])

  const aggregated = aggregateResults(results)
  await sheetWriter.writeAll(aggregated)
  await sheetWriter.writeRunLog(runId, runStart, aggregated, triggerType)
}
5.3 PRD pipeline logic
1. For each active feature with a PRD page ID:
   a. Confluence API → get page (storage format) → strip HTML → clean text
   b. Claude API (prdExtractor) → structured feature list as JSON
   c. Load previous snapshot from runState.json
   d. If no snapshot → baseline run, store it, no changelog entries
   e. If snapshot exists → diff new vs old
      - Added features   → changelog: type=scope_change, source=prd
      - Removed features → changelog: type=parking (presumed), source=prd
      - Modified         → changelog: type=scope_change, source=prd
   f. Store new snapshot with timestamp
5.4 UAT pipeline logic
1. For each active feature with a UAT sheet URL:
   a. Google Sheets API → all rows + headers
   b. Find all columns matching "UAT * Status" → pick highest number
   c. For each row: read Issue, latest UAT Status, Blocker column
      → apply rules (see Section 9)
      → if disposition != NO_ACTION → create changelog entry
   d. Return changelog entries + feature status updates
5.5 Slack pipeline logic
1. For each active feature:
   a. Load last_slack_run from runState.json
   b. For each channel:
      - conversations.history since last_slack_run
      - For messages with replies → conversations.replies (full thread)
   c. Pre-filter: discard threads with zero matching keywords
   d. Send qualifying threads to Claude API (classifier) in batch
   e. Parse JSON response, create changelog entries
   f. Update last_slack_run in runState.json
5.6 Sheet writer logic
Features tab (upsert):
  → Load all rows, index by Feature ID
  → If exists: update row in place. If new: append.

Scope Registry (upsert):
  → Index by (Feature ID + Task Name) composite key
  → Same upsert logic

Changelog (append with dedup):
  → Check source_message_id before writing
  → If already present → skip. Never update or delete.

Run Log (always append):
  → Never update. One row per run, always new.

UAT sheet rules logic
Column detection rule
1. Find all headers matching pattern: "UAT * Status" (case-insensitive)
2. Extract numeric suffix: "UAT 1 Status" → 1, "UAT 3 Status" → 3
3. Use column with highest number as active UAT Status column
4. If no numeric suffix, use last matching column by position
Status → disposition mapping
Latest UAT statusBlocker for release?DispositionTarget version
FailedNoPushed to next releaseCurrent version + 1 (e.g. V1 → V2)
FailedYesActive blockerCurrent version (flag, do not park)
Parked for laterAnyParkedUnversioned / TBD
PassedAnyNo action—
Passed with iterationAnyNo actionCurrent version (note iteration needed)
To be testedAnyNo action—

11
Error handling & resilience
Pipeline isolation
Each pipeline is in its own try/catch. Errors are caught, structured, and added to the run log. Orchestrator continues with remaining pipelines.
Partial Slack failure
If one channel fetch fails (e.g. bot not in channel), that channel is logged as failed. Remaining channels still run.
Retry logic
3 attempts per external API call. Exponential backoff: 1s → 2s → 4s. If all retries fail, error is logged and pipeline continues.
Deduplication
Before writing any changelog entry, system checks Source Message ID. If already present → skip. Re-runs never create duplicates.
Run log error format
[
  { "pipeline": "slack", "channel": "#scalper-dev", "error": "channel_not_found" },
  { "pipeline": "prd", "feature": "FEAT-001", "error": "confluence_api_timeout" }
]

13
Scheduling via cron-job.org
Setup steps
  1. Deploy Node.js server to Railway, Render, or Fly.io (all have free tiers)
  2. Note the public URL (e.g. https://scope-tracker.railway.app)
  3. Go to cron-job.org → Create new cronjob
  4. URL: https://scope-tracker.railway.app/run — Method: POST
  5. Schedule: 0 18 * * 1-5 (6pm every weekday)
  6. Enable "Save responses" to audit what each run returned

Claude Code task groups
Instructions to Claude Code
Create task-tracker.json at project root on first run. Read it at the start of every session to find the next pending group. Complete all tasks in one group, update statuses, print ✅ Group [N] — [Name] completed, then stop. On next session, resume from next incomplete group.
